Bug: <fx-update> orphans the modelItem it was triggered from

An fx-control with an <fx-update event="value-changed"> child (the documented way to trigger a full rebuild + recalculate + revalidate whenever a value changes) only reacts to the first value change. After that, any UI element bound to that same node — the control itself, or any other fx-output/fx-control pointed at it — silently stops re-rendering, forever. No error, no warning. This is not a whole-page freeze: other modelItems (even ones recalculated fresh on every keystroke, like the character-count example below) keep updating correctly the entire time.

Root cause: <fx-update> calls model.updateModel(), which runs rebuild(). rebuild() throws away mainGraph and rebuilds modelItems from scratch, retargeting existing ModelItems by path where possible (see the _prevModelItemsByPath/retarget() logic in fx-model.js). This retarget runs while the very node whose change triggered the update is mid-flight — and whatever list of bound UI elements/observers that node's ModelItem carries doesn't survive the retarget intact. Every other ModelItem, created fresh during that same rebuild with no such concurrent write in flight, keeps working fine.
