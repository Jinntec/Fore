Fixed: <fx-update> no longer orphans the modelItem it was triggered from

An fx-control with an <fx-update event="value-changed"> child (the documented way to trigger a full rebuild + recalculate + revalidate whenever a value changes) used to only react to the first value change. After that, any UI element bound to that same node — the control itself, or any other fx-output/fx-control pointed at it — silently stopped re-rendering, forever. No error, no warning. This was not a whole-page freeze: other modelItems (even ones recalculated fresh on every keystroke, like the character-count example below) kept updating correctly the entire time, which is what made the bug easy to miss.

Root cause: rebuild() in fx-model.js only re-registers ModelItems that go through an <fx-bind>'s init(). A ModelItem lazily created for a plain fx-control/fx-output ref with no <fx-bind> of its own — like query below — never goes through that path, so it was silently dropped from modelItems on the very next rebuild(). The UI elements kept their reference to the ModelItem, but that ModelItem was no longer recalculated or notified, so its observers never fired again.

Fix: rebuild() now reclaims any such lazily created ("synthetic") ModelItem whose backing node is still attached to its instance document, re-adding it to modelItems instead of silently discarding it.
